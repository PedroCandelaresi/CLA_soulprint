## B) HALLAZGOS CRÍTICOS

### 🔴 **HALLAZGO 1: Variables NEXT_PUBLIC hardcodeadas en build-time**

- **Severidad:** 🔴 CRÍTICA
- **Archivo(s):** 
  - Dockerfile
  - client.ts
  - docker-compose.yml

- **Evidencia:**
  ```dockerfile
  # Dockerfile storefront, líneas 18-27
  ARG NEXT_PUBLIC_VENDURE_API_URL
  ARG NEXT_PUBLIC_SITE_URL
  ARG NEXT_PUBLIC_REQUIRE_CUSTOMER_VERIFICATION
  ENV NEXT_PUBLIC_VENDURE_API_URL=$NEXT_PUBLIC_VENDURE_API_URL
  ENV NEXT_PUBLIC_SITE_URL=$NEXT_PUBLIC_SITE_URL
  ENV NEXT_PUBLIC_REQUIRE_CUSTOMER_VERIFICATION=$NEXT_PUBLIC_REQUIRE_CUSTOMER_VERIFICATION
  ```

  ```typescript
  // client.ts línea 56-70: Browser fallback uses baked-in value
  function resolveVendureApiUrl(): string {
      const publicApiUrl = readEnv('NEXT_PUBLIC_VENDURE_API_URL'); // ← undefined en runtime
      if (typeof window !== 'undefined') {
          return publicApiUrl || DEFAULT_BROWSER_VENDURE_API_URL; // ← /api/shop fallback
      }
      // ...
  }
  ```

- **Explicación:**
  - Con `output: "standalone"` en next.config.ts, Next.js compila todo en build-time
  - Las variables `ARG` se usan solo en build, los `ENV` que las referencian quedan compiladas en el bundle JavaScript
  - Al hacer `docker compose up` sin `--build`, Docker reutiliza la imagen vieja con URLs antiguas
  - El docker-compose.yml intenta override estas variables en runtime (línea 109), pero es **inútil** - ya están baked-in

- **Riesgo real:**
  - Cambias `/shop-api` a `https://new-vendure-instance.com/shop-api` en variables de entorno
  - Haces `docker compose up` (sin rebuild)
  - El frontend sigue apuntando a la API vieja
  - Los productos ven datos de **cuando se compiló la imagen** (pueden ser semanas viejos)

- **Solución recomendada:**
  - **Opción 1 (Recomendada para ahora):** Siempre hacer rebuild explícito: `docker compose up -d --build`
  - **Opción 2 (Mejor arquitectura):** Cambiar a runtime env via servidor Node o usar variables que lean de la red
  - **Opción 3 (Futuro):** Usar Next.js middleware para redirigir `/api/shop` dinámicamente según ambiente

- **Cambios necesarios después:** 
  - Crear script de deploy que siempre haga rebuild
  - Documentar en README que cambiar variables **requiere rebuild**

## CONCLUSIONES

**CLA Soulprint está funcional pero requiere atención INMEDIATA en dos áreas:**

- Semana 1: Fase 1 (crítica)
